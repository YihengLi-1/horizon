# SIS 自动化推进报告（2026-03-05 09:02:51 MST）

## 1. 本轮变更
- 变更范围严格控制为：`1` 个后端/运维硬化 + `1` 个前端修复。

### 后端/运维硬化（1/1）
- 文件: `scripts/readiness-check.sh`
- 改进:
  - 新增 CSRF 配置一致性校验函数：`read_env_value`、`check_env_pair_match`。
  - 在 `.env` 与 `.env.example` 中校验：
    - `CSRF_COOKIE_NAME` vs `NEXT_PUBLIC_CSRF_COOKIE_NAME`
    - `CSRF_HEADER_NAME` vs `NEXT_PUBLIC_CSRF_HEADER_NAME`
  - 新增 readiness 断言：`apps/web/lib/server-api.ts` 必须使用可配置 CSRF 名称（避免硬编码回归）。
- 价值:
  - 将“前后端 CSRF 命名漂移”提前到发布前检查，减少生产 403 风险。

### 前端修复（1/1）
- 文件: `apps/web/lib/server-api.ts`
- 问题:
  - Server-side API 调用路径仍硬编码 `csrf_token` / `x-csrf-token`，与环境变量配置化策略不一致。
- 修复:
  - 改为读取：`NEXT_PUBLIC_CSRF_COOKIE_NAME` / `NEXT_PUBLIC_CSRF_HEADER_NAME`。
  - 提供回退：`CSRF_COOKIE_NAME` / `CSRF_HEADER_NAME`，默认仍为 `csrf_token` / `x-csrf-token`。
- 价值:
  - SSR 场景与浏览器端 CSRF 命名完全对齐，避免仅服务端路径出现 token 校验失败。

## 3. 遗留风险
- 运行态风险: 未完成 `/health`、登录与关键业务接口在线验证，生产行为仍缺少本轮证据闭环。
- 配置风险: 当前 `.env` 未显式配置成对 CSRF 命名变量，虽有默认值兜底，但跨环境时仍可能漂移。
- 运维风险: 本环境无法稳定拉起本地 API/Web（本轮尝试中出现 watch/端口权限类限制），影响自动化 smoke 覆盖率。

## 4. CTO 视角问题与改进
- 问题: 安全配置策略已配置化，但发布门禁此前未强制校验跨端一致性，存在“可配置但不可验证”缺口。
- 本轮改进: 将 CSRF 命名一致性纳入 readiness 检查，形成可执行门禁。
- 建议:
  - 在 CI 中将 readiness 警告升级策略明确化（安全项建议 fail-fast）。
  - 在可用环境补齐运行态证据（health + API P0 e2e + web smoke）后再推进下一批变更。

## 5. 用户视角问题与改进
- 问题: 当 CSRF 命名在不同环境不一致时，用户会在部分操作中遭遇 403，感知为“偶发失败”。
- 本轮改进: 统一 SSR 与前端请求的 CSRF 命名来源，降低用户侧随机失败概率。
- 建议:
  - 对 `CSRF_TOKEN_INVALID` 在 UI 层统一提示“安全令牌过期，请重试”，并记录埋点便于运营追踪。
