# SIS 自动化推进报告（2026-03-05 04:02 MST）

## 本次变更

### 后端硬化（P1 运维）
- 新增可执行演练脚本：`scripts/backup-restore-drill.sh`
  - 执行全流程：`pg_dump` 备份 -> 创建演练库 -> `pg_restore` 恢复 -> 表数量与用户表抽样校验 -> 自动清理演练库。
  - 产出工件落盘到 `reports/sis-automation/artifacts`，输出大小与 SHA-256（环境支持时）。
  - 支持环境变量控制（host/port/user/db、是否保留演练库等），默认连接 `localhost:5432/sis_db`。
- 接入命令：`package.json` 新增 `pnpm backup:restore:drill`。
- 就绪检查增强：`scripts/readiness-check.sh` 新增“Backup restore drill script present”检查项。

### 前端修复（P1 监控可观测）
- `apps/web/app/admin/dashboard/page.tsx` 对齐后端 `/ops/metrics` 最新结构。
- 新增 Admin Dashboard 的 `Security Alerts` 区块：展示 `critical/warning` 告警、阈值与当前值。
- 新增安全计数卡片：
  - `CSRF Origin Blocked`
  - `CSRF Token Invalid`
  - `Login Rate Limited`
  - `Login Failed`

- 备份恢复演练脚本已落地，但当前环境未满足执行前置（Postgres 客户端工具 + 可连接数据库），演练尚未完成实证。
- 监控告警前端展示已完成，但仍需真实流量下验证告警触发与清除行为。

## CTO 视角：问题与改进
- 问题：生产化门槛已从“代码缺失”转为“运行环境不可复现”，CI/CD 前缺少可重复的最小验收环境。
- 改进：
  - 固化一套可运行的验收环境（至少含可连通 Postgres 与 API/Web 端口）。
  - 将 `backup:restore:drill` 纳入固定排程/流水线，并沉淀最近一次演练证据（备份文件摘要、恢复校验结果、耗时）。
  - 在告警面板基础上，补充值班动作 Runbook（触发阈值后 15 分钟内处置流程）。

## 用户视角：问题与改进
- 问题：当前用户无法获得“服务可用”保障，登录/关键页面可能直接不可用。
- 改进：
  - 先确保 API 可用性（数据库连通）再推广前端新监控视图。
  - 对前端关键提交路径继续强化失败提示（例如引导刷新会话/重试）并观测实际失败率下降。

## 阻塞原因与所需输入
- 阻塞原因：
  - 当前执行环境无法访问 Docker socket。
  - 本地无 `pg_dump/pg_restore/psql` 客户端工具。
  - API 因 `localhost:5432` 不可达无法启动。
- 所需输入（任一可行）：
  1. 提供可访问的 Postgres 实例连接信息（host/port/db/user/password）。
  2. 或提供可运行 Docker 的执行环境权限。
  3. 并安装 PostgreSQL 客户端工具（至少 `pg_dump`、`pg_restore`、`psql`）。
