## 变更

### 后端硬化（P1 安全/运维）
- 文件: `apps/api/src/main.ts`
- 目标: 监控告警增强（安全事件可观测 + 阈值告警）
- 实施内容:
  - 在运行时指标中新增安全计数: `csrfOriginBlocked`、`csrfTokenInvalid`、`loginRateLimited`。
  - 在 CSRF 中间件中对“来源被拦截/Token 无效”进行显式计数。
  - 在请求收尾统计中新增登录限流事件统计（`POST /auth/login` + `429`）。
  - 扩展 `/ops/metrics` 返回:
    - `security`: 安全计数聚合（含 `loginFailed` 从审计动作计数聚合）。
    - `alerts`: 基于阈值的告警数组（`CSRF_ORIGIN_BLOCK_SPIKE`、`CSRF_TOKEN_INVALID_SPIKE`、`LOGIN_RATE_LIMIT_SPIKE`、`HTTP_ERROR_RATE_HIGH`）。
    - `thresholds`: 当前生效阈值，支持环境变量覆盖。

### 前端修复
- 文件: `apps/web/lib/api.ts`
- 目标: 降低 CSRF token 缺失/过期导致的用户侧 403 抖动
- 实施内容:
  - 非安全方法（POST/PATCH/PUT/DELETE）发送前，若缺失 `csrf_token`，自动请求 `/auth/csrf-token` 并补发 header。
  - 若首次请求返回 `403` 且错误码 `CSRF_TOKEN_INVALID`，自动刷新 token 并仅重试一次。
  - 修正默认 `Content-Type` 设置逻辑：仅在存在 body 且非 `FormData` 时补 `application/json`，避免上传场景误设。

## CTO 视角问题与改进
- 问题: 生产化推进被环境依赖卡住，CI/CD 与本地执行缺少“可降级验证策略”。
- 改进:
  - 将 `/ops/metrics` 输出纳入告警管道（例如采集器 + 阈值告警），并把阈值配置纳入环境基线。
  - 增加无 DB 的最小可运行探针（如配置检查、依赖健康分级），减少完全阻塞。
  - 在 CI 增设“安全指标结构契约测试”，防止监控字段回归。

## 用户视角问题与改进
- 问题: 用户在 token 过期/丢失时可能遇到突发 403，体验为“操作偶发失败”。
- 改进:
  - 已实现客户端自动补 token + 一次重试，降低无感失败率。
  - 建议后续补充统一错误提示文案（例如“安全令牌已刷新，请重试”）与埋点，便于客服定位问题。
